# Vantage — Claude Code Instructions

Vantage is a fictional digital agency portfolio project demonstrating senior-level frontend architecture
using Next.js 15 + Payload CMS 3 across 3 distinct client verticals.

**Goal:** Showcase component ownership, design system thinking, headless CMS modeling,
and accessibility discipline at a senior frontend engineering level.

## GitHub Workflow — Required for Every Task

Every piece of work follows this exact flow. No exceptions.

```bash
# 1. Create issue (or use gh-task.ts)
gh issue create --title "feat: [name]" --label [component|feature|bug|collection|chore]

# 2. Create branch from the issue number
git checkout -b feat/[issue-slug]

# 3. Work on the task

# 4. Commit + push
git add [specific files]
git commit -m "feat: [description]"
git push origin feat/[issue-slug]

# 5. Open PR linked to the issue
gh pr create --title "feat: [name]" --body "closes #[issue-number]"

# 6. Merge + auto-close
gh pr merge --squash --delete-branch
```

Use `pnpm gh-task` (tools/scripts/gh-task.ts) to automate steps 1-2 and 5-6.

**Branch naming:**
- `feat/[slug]` — new feature or component
- `fix/[slug]` — bug fix
- `chore/[slug]` — setup, config, tooling
- `docs/[slug]` — documentation only

## @vantage/ui — Component Conventions

### File structure (every component)

```
packages/ui/src/components/[ComponentName]/
├── [ComponentName].tsx        ← semantic, React 19, Server Component when applicable
├── [ComponentName].stories.tsx← Default + all variants + all 3 themes
├── [ComponentName].test.tsx   ← Vitest + RTL
└── index.ts                   ← re-export
```

### What belongs in @vantage/ui vs apps/[site]/src/components

**Goes in @vantage/ui:** primitives usable across insurance, biotech, AND charity without feeling wrong.
Examples: Button, Card, Input, Badge, Modal, Hero, Section, NavBar, Footer, DataTable, Tabs.

**Stays local:** vertical-specific components tied to one domain.
Examples: QuoteForm (anchor), ClinicalTrialCard (veros), DonationCheckout (ripple).

### Theme system

Each app applies its theme via a CSS class on the root element:

```css
/* packages/ui/src/themes/anchor.css — navy, professional, insurance */
/* packages/ui/src/themes/veros.css  — clinical, white/green, biotech */
/* packages/ui/src/themes/ripple.css — warm, coral, charity */
```

All components use CSS variables — never hardcoded colors. Components are theme-agnostic.

### React 19 patterns

- Prefer Server Components. Use `"use client"` only when needed (event handlers, hooks, browser APIs).
- Use `use()` for async data in client components.
- Use `useActionState` + `useFormStatus` for forms + Server Actions.
- Use `useOptimistic` for optimistic updates.

## Payload CMS — Collection Conventions

Collections are namespaced by client site:

```
apps/cms/src/collections/
├── anchor/   → CoveragePlans, Testimonials, Agents, Claims, Pages
├── veros/    → ClinicalTrials, Publications, PatientResources, Pages
├── ripple/   → Campaigns, Donations, DonorStories, Pages
└── shared/   → Media, Users
```

After modifying any collection, run:
```bash
pnpm --filter=cms generate:types
```

This outputs updated types to `packages/payload-types/src/types.ts`.
Never manually edit generated types.

## Storybook Conventions

Every component requires stories for:
1. Default state
2. All prop variants
3. Interactive states (hover, focus, disabled, error, loading)
4. All 3 themes (anchor, veros, ripple) using the theme switcher
5. A11y check (the addon runs automatically)

```tsx
// [ComponentName].stories.tsx structure
export default { component: ComponentName, tags: ['autodocs'] }
export const Default: Story = { ... }
export const Variants: Story = { ... }
export const AnchorTheme: Story = { decorators: [withAnchorTheme] }
export const VeroTheme: Story = { decorators: [withVeroTheme] }
export const RippleTheme: Story = { decorators: [withRippleTheme] }
export const Interactive: Story = { play: async ({ canvasElement }) => { ... } }
```

## Playwright Conventions

```
apps/[site]/e2e/
├── flows/         → happy path e2e (one file per feature flow)
├── a11y/          → axe checks at each step of critical flows
└── visual/        → screenshot comparison
```

Use shared helpers from `tools/playwright/`:
- `axe-helper.ts` → `checkA11y(page, context)` — wraps axe-core
- `theme-helper.ts` → applies correct theme class per app

A11y tests run on every step of forms and checkout flows, not just the final page.

## A11y Requirements

| Site | WCAG Level | Notes |
|------|-----------|-------|
| Anchor | AA | Complex forms (quote flow), payment |
| Veros | AA | Vulnerable users, regulatory context |
| Ripple | AA | Payment flow, diverse users |

- Use React Aria for all interactive primitives (not Radix)
- Every form field needs proper `label`, `aria-describedby` for errors
- Focus management on modals and multi-step flows
- Test with keyboard-only navigation in Playwright

## Client Sites — Domain Context

### Anchor (Car Insurance)
- Inspiration: Old Republic, Progressive, Geico
- Design: Navy blue, serif headings, trustworthy, fintech-clean
- Critical flows: Quote (3-step form), Coverage plans comparator, Claims portal
- A11y: High — complex forms, payment flow

### Veros (Biotech / Clinical)
- Inspiration: Data Allergan, Pfizer, clinical research sites
- Design: White/green, clinical, scientific, high contrast
- Pages: Clinical trials portal, Research publications, Patient resources
- A11y: Very high — vulnerable users, regulatory requirements

### Ripple (Charity + Donations)
- Inspiration: ChangeUp (checkout donations + Discord bot)
- Design: Warm coral, rounded, human, emotional
- Pages: Campaign pages, Donation checkout, Discord bot widget
- A11y: High — payment flow, diverse users
